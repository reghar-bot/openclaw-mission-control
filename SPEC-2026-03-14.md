## First-Principles Analysis

### What does Ragnar need to know at a glance?

Three questions, in order of urgency:

1. **Is the system healthy right now?** — Gateway running, crons firing, no cascade failures.
2. **What broke since I last looked?** — Which crons are failing, how many consecutive errors, what is the error.
3. **Is it costing what it should?** — Budget burn rate vs. remaining days in month.

Everything else is secondary. Pipeline step details, agent performance trends, session counts — these are useful but not glance-level. They belong behind a scroll, not above the fold.

### What is noise?

- Individual cron schedules (42+ crons — listing every schedule is a wall of text)
- Agent file sizes (audit concern, not dashboard concern)
- Memory line counts (internal hygiene)
- Proposal status (workflow detail, not health signal)
- Session counts (only relevant when above threshold)

### What is the MINIMUM set?

1. One aggregate health status (HEALTHY / DEGRADED / CRITICAL)
2. List of currently-failing crons (consecutiveErrors >= 1) with error category
3. Budget: spent / limit / projected, with a visual burn indicator
4. Pipeline status for the 3-4 active pipelines
5. Last 10 cron runs (activity heartbeat — proof the system is alive)

That is five widgets. Everything else must justify its screen space.

## Architecture Decision: Data Flow

### Option A: Static HTML generated from local JSON (CHOSEN)

A Python script runs locally every 10 minutes via OpenClaw cron. It calls `openclaw cron list --json`, reads cost data from the existing daily cost tracker, and writes a `data.json` file. A second step pushes `data.json` to a GitHub repo. Cloudflare Pages serves the static HTML, which fetches `data.json` on load and on a 5-minute interval.

**Why this approach:**
- No backend needed. No API endpoint to build or maintain.
- OpenClaw already runs on Ragnar's Mac mini 24/7 — a cron that writes JSON is trivial.
- Cloudflare Pages serves globally with zero ops.
- The dashboard is read-only by design — it reads a JSON file, nothing more.

**Data pipeline:**
```
[Mac mini] openclaw cron (*/10)
  → runs generate-data.sh
  → calls openclaw cron list --json
  → reads cost tracker files
  → reads SYSTEM-HEALTH.md (when it exists)
  → writes data.json
  → git commit + push to github repo
  → Cloudflare Pages auto-deploys
  → dashboard fetches data.json on load + every 300s
```

### Why not a JSON API endpoint?

Building an API means Kodex has to create and maintain a server process. The Mac mini already runs OpenClaw's gateway. Adding another process increases the ops surface for zero benefit — the data changes at most every 10 minutes, so polling a static file is equivalent to polling an API.

## Password Protection

**Approach:** Client-side SHA-256 hash check. On load, the page shows a password input. The entered password is hashed with SHA-256 in the browser and compared against a hardcoded hash. If it matches, the dashboard renders. The password and hash never leave the browser.

**Implementation:**
```javascript
const HASH = '<sha256 of chosen password>';
async function checkPassword(input) {
  const encoded = new TextEncoder().encode(input);
  const hashBuffer = await crypto.subtle.digest('SHA-256', encoded);
  const hashHex = Array.from(new Uint8Array(hashBuffer))
    .map(b => b.toString(16).padStart(2, '0')).join('');
  return hashHex === HASH;
}
```

Store the authenticated state in `sessionStorage` so it persists across page refreshes but not across tabs/sessions. This is intentionally weak security — it is a personal tool, not a bank vault. The goal is to prevent a casual visitor from seeing system internals, not to resist a determined attacker.

**Password:** Ragnar picks the password. Kodex hashes it and embeds the hash.

## Widget Specifications

### Widget 1: Aggregate Health Status

**What:** Single large status indicator at the top. One word: HEALTHY, DEGRADED, or CRITICAL. Color-coded background.

**Why it earns its place:** This is the single most important piece of information. Ragnar opens the dashboard, sees green or red, and knows whether to dig deeper.

**Logic:**
- HEALTHY (green): 0 crons with consecutiveErrors >= 2, gateway not reporting errors
- DEGRADED (amber): 1-3 crons with consecutiveErrors >= 2, OR budget > 85% at day < 25
- CRITICAL (red): 4+ crons with consecutiveErrors >= 2, OR budget > 95%, OR gateway down

**Data source:** Computed client-side from `data.json` fields.

### Widget 2: Health Summary Bar

**What:** Four compact metric cards in a horizontal row.

| Card | Value | Source |
|------|-------|--------|
| Crons OK | `{healthy}/{total}` | Count crons where consecutiveErrors == 0 |
| Gateway | `OK` or `DOWN` | `data.json.gateway.status` |
| Budget | `${spent}` | `data.json.budget.currentMonth` |
| Errors | Count of crons with consecutiveErrors >= 1 | Computed from cron data |

**Why it earns its place:** Four numbers that together tell the full health story. Gateway confirms the runtime is alive. Budget confirms cost control. Cron ratio and error count confirm workload execution.

**Data source:** All from `data.json`.

### Widget 3: Failing Crons Table

**What:** Table showing ONLY crons with consecutiveErrors >= 1. Sorted by error count descending. Columns: Name, Agent, Errors, Last Error (truncated), Error Category, Last Run.

**Why it earns its place:** This is the action list. If everything is healthy, this section shows "All crons healthy" in muted text and takes up one line. If things are broken, this is where Ragnar looks.

**Conditional visibility:** If zero failing crons, collapse to a single "All clear" line with a checkmark.

**Data source:** Filter `data.json.crons` where `consecutiveErrors >= 1`.

**Error category mapping** (parsed client-side from `lastError` string):
- Contains "channel" or "recipient" → `CONFIG`
- Contains "timed out" or "timeout" → `TIMEOUT`
- Contains "message" or "delivery" → `DELIVERY`
- Contains "write" or "tilde" or "path" → `FILE_WRITE`
- Contains "credit" or "balance" → `PROVIDER`
- Else → `OTHER`

### Widget 4: Budget Burn

**What:** Progress bar showing month-to-date spend vs. $250 budget. Below: day of month, projected end-of-month spend, status label.

**Why it earns its place:** The chief-of-staff analysis found budget monitoring is passive — $235/$250 with no alarm. This widget makes the budget viscerally visible. A red progress bar at 94% is impossible to ignore.

**Logic:**
- `projection = (spent / dayOfMonth) * daysInMonth`
- ON TRACK: projection <= budget
- ELEVATED: projection > budget but < budget * 1.15
- OVER BUDGET: projection > budget * 1.15

**Data source:** `data.json.budget` object. The generate script reads from `/Users/reghar/.openclaw/workspace/logs/costs/` (existing daily cost tracker files) and computes the month total.

### Widget 5: Pipeline Status

**What:** Four rows, one per known pipeline. Each shows: pipeline name, status dot (green/red/amber/gray), last successful run date, and a one-line status note.

**Pipelines to track:**

| Pipeline | Cron IDs to monitor | Status logic |
|----------|-------------------|--------------|
| Champagne Brief (Mon) | Research `b786cb07`, Curation, Brief, Judge, EIC, Send | RED if any step has consecutiveErrors >= 2 on most recent Monday |
| Automation Scout (M/W/F) | Research Part 1+2, Report | GREEN if last run was within expected schedule |
| Nightly Maintenance | Audit, Memory Consolidation, QMD Re-index | GREEN if all ran OK in last 24h |
| RE Daily/Weekly | RE crons | GRAY (paused) — hardcoded until re-enabled |

**Why it earns its place:** The overnight audit found the Champagne pipeline is the single biggest reliability problem. Pipeline-level status (not individual cron status) is what matters for "is the product being delivered."

**Data source:** `data.json.pipelines` — the generate script groups crons by pipeline and computes aggregate status.

### Widget 6: Activity Feed

**What:** Last 10 cron runs, most recent first. Each row: timestamp (HH:MM), cron name, status badge (ok/error/skipped), duration.

**Why it earns its place:** This is the heartbeat. A stopped activity feed means the system is dead. A feed full of errors means it is sick. A feed with regular "ok" entries means it is alive. This answers "when did it last do something?" at a glance.

**Data source:** `data.json.crons` sorted by `lastRunAtMs` descending, take top 10.

### Widget 7: Cron Fleet Grid

**What:** All crons as small tiles in a compact grid. Each tile is ~80px wide, shows a colored dot (status), abbreviated name (first 20 chars), and a tooltip with full details on hover. Tiles grouped by agent, with agent name as a section header.

**Why it earns its place:** After seeing the summary, Ragnar may want to scan the full fleet. This grid lets him visually spot clusters of red/amber without reading a table. It is below the fold — you scroll to it only if you want detail.

**Data source:** All entries in `data.json.crons`.

## Generate Script: `scripts/generate-data.sh`

Location: `/Users/reghar/.openclaw/workspace/mission-control/scripts/generate-data.sh`

**What it does:**
1. Runs `openclaw cron list --json` and captures output
2. Reads the latest cost file from `/Users/reghar/.openclaw/workspace/logs/costs/`
3. Checks gateway status via `pgrep -f "openclaw gateway"` (or reads gateway PID file)
4. Reads `SYSTEM-HEALTH.md` if it exists (future — chief-of-staff analysis recommends creating this)
5. Pipes everything through a Python script that produces `data.json`
6. Commits and pushes to the GitHub repo

**Cron entry:**
```
Schedule: */10 * * * *
Name: Mission Control — Data Refresh [exec]
Payload: exec, command: bash /Users/reghar/.openclaw/workspace/mission-control/scripts/generate-data.sh
Timeout: 60s
Delivery: none
```

This is a lightweight exec cron — no LLM calls, no tokens burned. Pure shell + Python data transformation.

## What Is Explicitly Out of Scope

These items were considered and rejected for v1:

| Feature | Why excluded |
|---------|-------------|
| Actions/buttons (restart cron, clear errors) | Constraint: read-only dashboard. Actions happen through OpenClaw CLI or Telegram buttons. |
| Historical charts (cost over time, error trends) | Requires storing time-series data. v1 is a snapshot, not a time machine. Add in v2 if needed. |
| Agent performance scores | No quality scoring infrastructure exists yet (chief-of-staff analysis recommends building it). Dashboard can show this once the data exists. |
| Session management | Session count is shown as a number. Managing sessions is an ops action, not a dashboard feature. |
| Notification/alerting | Telegram and Discord already handle alerts. The dashboard is for pull-based observation, not push-based alerting. |
| 24h timeline view | The BRIEF.md proposed this. Cut for v1 — the activity feed serves the same purpose with less complexity. Re-evaluate for v2. |
| Auto-refresh via WebSocket | Static file on Cloudflare Pages. The 5-minute meta-refresh + fetch is sufficient for a system that updates every 10 minutes. |
